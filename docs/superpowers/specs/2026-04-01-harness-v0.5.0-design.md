# sps-harness v0.5.0 — Best Practice Adoption

v0.5.0 将 Claude Code 官方文档和 shanraisshan/claude-code-best-practice 仓库中验证有效的机制引入 sps-harness，提升 skill 运行效率、agent 跨会话能力和配置一致性。

## 背景

调研 Claude Code 官方文档（docs.anthropic.com）和社区 best-practice 仓库（26.9K star）后，识别出 6 项可直接落地的改进。这些改进都是 Claude Code 原生支持的机制，只是 harness 尚未使用。

**不做的：**
- `<important if="...">` 条件标签：harness 已有升级链（规则→hook）解决同一问题，加条件标签是冗余
- Agent Teams：需要 tmux + iTerm2，对目标用户门槛过高
- 跨模型 QA（Codex 审查）：增加依赖，插件场景不适用
- /loop 集成：hooks 已覆盖持续监控

## 1. Agent Memory — security-reviewer 跨会话记忆

### 是什么

给 security-reviewer agent 添加 `memory: project` 前置字段，使其获得独立的持久记忆目录 `.claude/agent-memory/security-reviewer/`。

### 为什么

当前 security-reviewer 每次从零开始扫描。对于已审计过多次的项目，它会重复报告同样的 false positive（比如测试文件里的 mock API key），用户需要反复解释"这个可以忽略"。跨会话记忆让 agent 记住已确认的 false positive、历史发现和攻击面基线。

### 怎么做

**agents/security-reviewer.md frontmatter 改动：**

```yaml
---
name: security-reviewer
memory: project        # 新增
model: sonnet
maxTurns: 30
allowedTools:
  - Read
  - Glob
  - Grep
---
```

**agents/security-reviewer.md 正文新增段落（在 Scan Procedure 之前）：**

```markdown
## Memory Usage

Before scanning, read your memory to check for:
- Previously confirmed false positives (skip these in the report, note them as "Known — accepted")
- Last audit findings (compare and mark each as "New", "Fixed", or "Still present")
- Known attack surface baseline (highlight changes since last audit)

After scanning, write a brief summary to memory:
- Date and scope of this audit
- New findings with severity
- Any items the user explicitly marked as false positive during this session
```

**选 `project` 的原因：** 存储在 `.claude/agent-memory/security-reviewer/`，可提交到 git 共享团队安全审计历史。`user` 只存本地不共享，`local` 也不共享且不适合插件场景。

## 2. context: fork + On-demand Stop Hook — verification-loop

### 是什么

给 verification-loop 添加 `context: fork`（在隔离子窗口运行）和 skill 级 Stop hook（强制要求贴证据）。

### 为什么

verification-loop 执行 6 个阶段，每个阶段运行命令并产生终端输出。这些中间输出对用户没有持续参考价值（用户只关心最终的 PASS/FAIL 表格），但会大量占用主上下文窗口。fork 后主对话只收到最终汇总。

Stop hook 防止 Claude 跳过某个阶段直接标 PASS（这在长上下文中偶尔发生）。

### 怎么做

**skills/verification-loop/SKILL.md frontmatter 改动：**

```yaml
---
name: verification-loop
description: (keep existing description unchanged)
context: fork
hooks:
  Stop:
    - type: prompt
      prompt: >
        Before reporting results, confirm: every PASS phase has the actual
        command output quoted as evidence. No phase can be marked PASS without
        showing the real terminal output.
---
```

SKILL.md 正文不需要改动。现有的 6 阶段流程保持原样，只是运行环境从主上下文变成隔离子窗口。

### 行为变化

- 用户体验：执行 verification-loop 后，主对话收到一条汇总消息（PASS/FAIL/SKIP 表格 + 失败详情），中间过程不可见
- 用户不能对中间步骤追问（trade-off：已确认接受）
- Stop hook 只在 verification-loop 运行期间生效，不影响其他 skill

## 3. 动态 Shell 注入 — harvest & harness-audit

### 是什么

在 SKILL.md 中使用 `` !`command` `` 语法，让 Claude 加载 skill 时自动执行 shell 命令并将输出替换进上下文。

### 为什么

harvest 需要知道项目技术栈、有没有设计文档、CLAUDE.md 是否存在。harness-audit 需要知道 .claude/ 的基本骨架。当前两个 skill 都通过多次工具调用（Bash、Read、Glob）来探测这些信息，浪费 3-5 个工具调用和对应的 token。

动态注入在 skill 加载时一次性获取运行时数据，Claude 拿到的是预填充的项目快照。

### 怎么做

**skills/harvest/SKILL.md — 在 "Two modes" 段之前插入：**

```markdown
## 当前项目快照

- Git 分支: !`git branch --show-current 2>/dev/null || echo "not a git repo"`
- 包管理器: !`(ls pnpm-lock.yaml 2>/dev/null && echo pnpm) || (ls yarn.lock 2>/dev/null && echo yarn) || (ls package-lock.json 2>/dev/null && echo npm) || echo "unknown"`
- Node 版本: !`node -v 2>/dev/null || echo "N/A"`
- Python 版本: !`python3 --version 2>/dev/null || echo "N/A"`
- 设计文档: !`ls docs/superpowers/specs/*.md 2>/dev/null || echo "none"`
- 现有 CLAUDE.md: !`(test -f .claude/CLAUDE.md && echo "exists, $(wc -l < .claude/CLAUDE.md) lines") || echo "not found"`
```

**skills/harness-audit/SKILL.md — 在 "Two modes" 段之前插入：**

```markdown
## 当前 .claude/ 快照

- .claude/ 存在: !`test -d .claude && echo "yes" || echo "no"`
- CLAUDE.md: !`(test -f .claude/CLAUDE.md && echo "$(wc -l < .claude/CLAUDE.md) lines") || (test -f CLAUDE.md && echo "root, $(wc -l < CLAUDE.md) lines") || echo "missing"`
- rules 数量: !`ls .claude/rules/*.md 2>/dev/null | wc -l`
- agents 数量: !`ls .claude/agents/*.md 2>/dev/null | wc -l`
- hooks 配置: !`test -f .claude/hooks.json && echo "exists" || (test -f hooks/hooks.json && echo "in hooks/") || echo "missing"`
- 设计文档: !`ls docs/superpowers/specs/*.md 2>/dev/null | wc -l`
```

### 设计约束

- 所有命令带 `2>/dev/null` 容错，不存在时输出明确的 fallback 文本
- 不读取敏感内容（不读 .env 值，不读文件正文，只读元信息）
- 命令在用户项目根目录执行（不是插件目录）
- Windows 兼容性：这些命令在 bash shell 下执行（Claude Code 在 Windows 也用 bash），ls/test/wc 均可用

## 4. description 字段优化

### 是什么

将 3 个 skill 的 description 从"功能摘要"改为"触发条件"格式，让 Claude 更准确地判断何时自动调用。

### 为什么

Thariq（Claude Code 团队）明确指出：description 是写给模型的触发器，不是给人读的摘要。以"Use when:"开头比以功能描述开头更有效。当前 product-drafter、product-launcher、prompt-audit、verification-loop 的 description 已经是触发格式，剩余 3 个需要对齐。

### 怎么做

**harvest — 重写 description：**

```yaml
description: >
  Use when: starting a new project after brainstorming, project has no .claude/CLAUDE.md,
  or design docs changed and CLAUDE.md needs refresh.
  Trigger on: "harvest", "/harvest", "生成CLAUDE.md", "generate project config",
  "sync design docs to claude", "把设计文档同步到.claude".
  Use after Superpowers brainstorming produces a design doc, before writing-plans.
  Also use on existing projects that lack .claude/CLAUDE.md.
  Do not trigger for: editing existing CLAUDE.md content (user does that manually),
  .claude/ health assessment (harness-audit's job).
```

**harness-audit — 重写 description 开头：**

```yaml
description: >
  Use when: user wants to know how well their .claude/ is set up, starting a new project
  and unsure what .claude/ should contain, or periodic health check after many changes.
  Trigger on: "harness audit", "审计harness", "check my harness", "评估.claude成熟度",
  "/harness-audit", "harness init", "初始化harness", ".claude健康检查", "harness体检",
  "我的harness怎么样", "score my harness", "给harness打个分", "check harness health",
  "how should I structure .claude/", "harness 成熟度", "harness 打分".
  Also trigger when the user asks to evaluate, score, or assess their .claude/ configuration health.
  Do not trigger when the user wants to create, modify, fix, or reorganize .claude/ files
  (that is harness-creator's job), review individual prompt files (prompt-audit's job),
  or configure settings.json hooks (update-config's job).
```

**strategic-compact — 重写 description：**

```yaml
description: >
  Use when: suggest-compact hook fires, user says /save-compact or "保存上下文",
  context feels long and you notice forgetting earlier conversation details,
  or recovering after a context reset.
  Do not trigger for: routine /compact without state saving,
  or when conversation is short and no state worth preserving.
```

## 5. settings.json + rules 去重

### 是什么

新建插件级 settings.json 放确定性权限拒绝规则，清理 rules 中与 hooks 重复的条目。

### 为什么

Boris Cherny（Claude Code 创造者）明确建议：能用 settings.json 做的确定性行为不要靠提示词。`permissions.deny` 在权限层直接拦截，Claude 连工具调用都发不出去，比 hook 更彻底。同时 security-standards.md 自带一条 TODO 标注了 hook 重复问题。

### 怎么做

**新建 settings.json（插件根目录）：**

```json
{
  "permissions": {
    "deny": [
      "Bash(git * --no-verify*)",
      "Bash(git push --force *)",
      "Bash(git push -f *)",
      "Bash(rm -rf *)"
    ]
  }
}
```

block-no-verify.js hook 保留，作为双重防线（hook 能输出解释信息告诉 Claude 为什么被拦，settings.json 只是静默拒绝）。

**rules/security-standards.md 改动：**

将已被 hook 自动化的两项从手动清单移出，改为引用说明：

```markdown
## Security Standards

Automated checks (commit-guard hook handles, no manual verification needed):
- Hardcoded secrets → commit-guard.js auto-scans on every commit
- console.log / debugger → commit-guard.js auto-detects on every commit

Before every commit, manually verify:
- [ ] All user inputs validated and sanitized
- [ ] Database queries use parameterized statements (SQL injection prevention)
- [ ] User-generated HTML is escaped (XSS prevention)
- [ ] CSRF protection enabled on state-changing endpoints
- [ ] Authentication and authorization verified on protected routes
- [ ] Rate limiting applied to public-facing endpoints
- [ ] Error messages expose only user-friendly descriptions, omitting stack traces, file paths, and version strings

Store secrets in environment variables or a secret manager. Validate required secrets are present at application startup, because missing secrets should fail loudly rather than silently.

Rotate any secret that may have been exposed immediately, because delayed rotation increases the window of vulnerability.

When a security issue is found: stop current work immediately, run /security-review, fix all CRITICAL issues before resuming.

<!-- Added: 2026-03-30 | Source: ECC security.md adapted for Superpowers ecosystem | Reason: 建立安全实践基线 -->
```

删除底部的 TODO 注释（本次改动完成了该 TODO）。

**rules/git-standards.md 微调：**

第一行加 hook 引用：

```markdown
Format commit messages as `<type>: <description>` (auto-validated by commit-guard hook), because consistent formatting makes history scannable and enables changelog generation.
```

其余内容保持不变。

## 风险和注意事项

1. **动态 shell 注入的 Windows 兼容性**：Claude Code 在 Windows 也使用 bash shell，ls/test/wc 命令可用。但如果用户安装了不标准的 bash 环境，某些命令可能行为不同。所有命令已带 `2>/dev/null` 容错。

2. **context: fork 的信息丢失**：verification-loop fork 后用户无法追问中间步骤。这是有意的 trade-off（已确认）。如果用户需要深入某个阶段的细节，可以单独跑对应的命令（如 `pnpm run build`）。

3. **Agent Memory 的存储增长**：security-reviewer 每次审计写入摘要，长期会积累。依赖 Claude 的自动记忆管理机制（MEMORY.md 前 200 行加载，超出后自动拆分主题文件）。

4. **settings.json permissions.deny 的覆盖范围**：插件的 settings.json 优先级低于项目的 `.claude/settings.local.json`。如果用户在本地设置中显式 allow 了被拒绝的命令，本地设置会覆盖。这是合理的（用户的显式意图优先于插件默认）。
