# superpowers-harness v0.2.0 设计文档

## 背景

superpowers-harness v0.1.0 提供了 .claude/ 架构的审计、安全审查和质量门禁能力。但用户反馈两个核心痛点：

1. **从零到一的断层**：Superpowers brainstorming 产出设计文档后，项目的 .claude/ 仍然是空的。设计文档里的架构信息、技术栈、约定没有沉淀到 CLAUDE.md 和 rules/ 里，导致后续开发时 Claude 每次都从零理解项目。

2. **缺少编程规范基线**：harness v0.1.0 的 3 条 rules 聚焦 harness 方法论本身（workflow-map、harness-method、noise-filter），没有通用编程规范（代码风格、测试标准、安全检查、git 约定）。Claude 写代码时缺少基本的质量约束。

调研了 ECC（Everything Claude Code，50K+ stars）的 77 条规则后发现：其中 5 条编程规范有参考价值，但 4 条是 ECC 专属流程指南（引用不存在的 agents），1 条内容过薄。直接安装会和 Superpowers 冲突。

同时发现 harness 的代码质量 hooks 存在空白：quality-gate 只跑 linter，缺少自动格式化、类型检查、commit 校验等确定性保障。Superpowers 的 TDD/code-review skills 是流程依赖的（不走完整流程就不触发），不能替代 hooks 的实时自动执行。

## 设计目标

superpowers-harness v0.2.0 新增三组能力，补齐「通用规范 → 项目配置 → 实时门禁」全链路：

1. **编程规范 rules**：4 个规则文件，装了 harness 就全局生效
2. **/harvest 命令**：从设计文档或项目文件自动生成 .claude/CLAUDE.md
3. **代码质量 hooks**：自动格式化、类型检查、commit 校验、密钥检测

```
装 superpowers-harness
  └─ rules/ 自动生效 → 通用编程规范（全局，所有项目）

Superpowers brainstorming → 设计文档
  └─ /harvest → 生成 .claude/CLAUDE.md（当前项目）

writing-plans → executing-plans
  └─ hooks 实时运行 → 格式化、类型检查、commit 校验（当前项目）
```

## 不做的事

- 不替代 Superpowers 的流程（brainstorming、writing-plans、executing-plans）
- 不替代 ECC（不提供 135 个 skills 或 30 个 agents）
- 不生成项目级 path-scoped rules（v0.2.0 只生成 CLAUDE.md，rules 留给后续版本）
- 不做语言专属规则包（v0.2.0 规则是语言无关的，语言细节由 /harvest 写入 CLAUDE.md）

## 一、编程规范 Rules

### 1.1 设计原则

- 参考 ECC 的编程规范精华，去除所有 ECC 专属引用
- 所有工具引用指向 Superpowers 和 harness 对应组件
- 遵循 harness-method.md 的规则编写标准：正面语言、包含理由、范围约束、来源标注
- 每个文件控制在 50 行以内（harness-method 的 File Placement Principles）

### 1.2 新增文件清单

在 `rules/` 目录新增 4 个文件：

#### rules/coding-standards.md

内容范围：
- 不可变优先：创建新对象而非修改现有对象，因为不可变数据避免隐藏副作用，便于调试和并发安全
- 小文件原则：每个文件 200-400 行，上限 800 行，超出则按功能拆分，因为小文件提高可读性和 Claude 的处理精度
- 函数聚焦：每个函数控制在 50 行以内，嵌套不超过 4 层，因为短函数更容易理解和测试
- 显式错误处理：在每个层级显式处理错误，UI 层提供用户友好的消息，服务端记录详细上下文，不静默吞掉错误
- 边界输入校验：在系统边界（用户输入、外部 API、文件内容）使用 schema 校验，快速失败并给出清晰错误消息

来源标注：`<!-- Added: 2026-03-30 | Source: ECC coding-style.md adapted for Superpowers ecosystem | Reason: 提供通用编码基线 -->`

#### rules/testing-standards.md

内容范围：
- TDD 流程：写测试（RED）→ 跑测试确认失败 → 写最少实现（GREEN）→ 跑测试确认通过 → 重构（IMPROVE）→ 验证覆盖率
- 覆盖率目标：80% 以上，因为低于此阈值时回归风险显著上升
- 测试类型：单元测试（函数、工具函数、组件）+ 集成测试（API 端点、数据库操作）+ E2E 测试（关键用户流程）
- 测试失败处理：检查测试隔离性 → 验证 mock 正确性 → 修实现而非修测试（除非测试本身有误）
- 工具引用：使用 Superpowers 的 test-driven-development skill 进行 TDD 流程

来源标注：`<!-- Added: 2026-03-30 | Source: ECC testing.md adapted for Superpowers ecosystem | Reason: 建立测试纪律基线 -->`

#### rules/security-standards.md

内容范围：
- 提交前安全检查清单：无硬编码密钥、所有用户输入已校验、SQL 注入防护（参数化查询）、XSS 防护（转义 HTML）、CSRF 保护已启用、认证授权已验证、接口有限流、错误消息不泄露敏感数据
- 密钥管理：使用环境变量或密钥管理器，启动时验证必要密钥存在，泄露时立即轮换
- 安全响应协议：发现安全问题时立即停止 → 运行 /security-review → 修复 CRITICAL 问题后再继续
- 工具引用：使用 harness 的 /security-review 进行安全审计

来源标注：`<!-- Added: 2026-03-30 | Source: ECC security.md adapted for Superpowers ecosystem | Reason: 建立安全实践基线 -->`

#### rules/git-standards.md

内容范围：
- Commit message 格式：`<type>: <description>`，type 包括 feat/fix/refactor/docs/test/chore/perf/ci
- PR 流程：分析完整 commit 历史（不只看最后一个 commit）→ 用 `git diff [base-branch]...HEAD` 查看所有变更 → 写完整的 PR 摘要 → 包含测试计划
- 分支策略：功能分支从 main 创建，完成后通过 PR 合入，不直接推 main

来源标注：`<!-- Added: 2026-03-30 | Source: ECC git-workflow.md adapted for Superpowers ecosystem | Reason: 统一 git 协作规范 -->`

## 二、/harvest 命令

### 2.1 核心功能

读取项目的设计文档和/或项目文件，自动生成 `.claude/CLAUDE.md`，让 Claude 在后续开发中具备项目上下文。

### 2.2 两种模式

**模式 A：有 Superpowers 设计文档**

优先读取 `docs/superpowers/specs/` 下的设计文档，提取：
- 项目名称和简述
- 技术栈（框架、数据库、API 等）
- 架构（目录结构、模块划分、数据流）
- 约定（命名、文件组织、代码风格偏好）

再扫描项目文件补充设计文档中未明确的信息：
- package.json / requirements.txt / go.mod → 依赖和脚本命令
- tsconfig.json / pyproject.toml → 语言配置
- 目录结构 → 验证和补充架构描述

**模式 B：无设计文档（回退模式）**

没有 `docs/superpowers/specs/` 时，纯粹扫描项目文件推断：
- package.json 的 scripts 字段 → 命令
- package.json 的 dependencies → 技术栈
- tsconfig.json → TypeScript 配置
- 目录结构（src/ 下的子目录）→ 架构
- .env.example 或 .env.local → 环境变量（不读取值，只读取 key 名称）

推断能力有限，生成的 CLAUDE.md 会标注 `<!-- Generated by /harvest (inferred mode). Review and refine. -->`。

### 2.3 生成的 CLAUDE.md 结构

```markdown
# [项目名称]

## 技术栈
[框架] + [数据库] + [关键依赖]

## 命令
[dev 命令]       # 开发服务器
[build 命令]     # 生产构建
[test 命令]      # 运行测试
[lint 命令]      # 代码检查

## 架构
[一段话描述整体架构]

[目录结构树，标注每个主要目录的职责]

## 约定
- [约定 1：文件组织规则]
- [约定 2：数据访问模式]
- [约定 3：其他项目特定约定]

<!-- Generated by /harvest from [source]. Last updated: YYYY-MM-DD -->
```

目标：控制在 100 行以内，符合 harness-audit Dimension 1 的 Check 1.1（CLAUDE.md 存在且简洁）。

### 2.4 实现方式

/harvest 实现为一个 skill（`skills/harvest/`），通过 command（`commands/harvest.md`）触发。

skill 的工作流程：

1. **检测模式**：检查 `docs/superpowers/specs/` 是否存在设计文档
2. **收集信息**：
   - 模式 A：读设计文档 + 扫项目文件
   - 模式 B：只扫项目文件
3. **检测现有 CLAUDE.md**：如果已存在，读取内容
4. **生成 CLAUDE.md**：按模板结构生成内容
5. **处理更新**：
   - 首次生成：直接写入 `.claude/CLAUDE.md`
   - 更新已有：展示 diff，让用户确认后写入
6. **验证**：运行 harness-audit 的 check 1.1 确认结果合规
7. **提交**：将生成的 CLAUDE.md 加入 git

### 2.5 更新策略

重跑 /harvest 时（项目已有 CLAUDE.md）：

- 重新读取设计文���和项目文件，生成新版本
- 和现有 CLAUDE.md 做对比���展示差异
- 用户选择：接受全部更新 / 逐段选��� / 取消
- 生成段落的识别方式：/harvest 生成的段落使用固定标题���技术栈、命令、架构、约定）。标题不在此列表中的段落视为用户手动添加，更新时保留
- 底部注释更新为最新生成时间：`<!-- Last updated: YYYY-MM-DD -->`

### 2.6 session-start 增强

修改 `hooks/scripts/session-start.js`，新增检测逻辑：

```

  → 提示 "Design docs found but no CLAUDE.md. Run /harvest to generate project configuration."
```

## 三、代码质量 Hooks

### 3.1 设计原则

- 复用现有的 `lib/utils.js` 工具函数
- 每个 hook 脚本独立，失败不阻断其他 hooks
- 动态检测可用工具（和 quality-gate.js 检测 linter 的方式一致）
- Windows 兼容（用 Node.js 而非 shell 脚本）

### 3.2 新增 hooks

#### 3.2.1 auto-format.js（PostToolUse，matcher: Edit|Write|MultiEdit）

编辑文件后自动格式化。

检测逻辑（按优先级）：
1. biome.json 存在 → `npx biome format --write <file>`
2. prettier 配置存在（.prettierrc / prettier.config.*）或 package.json 有 prettier 依赖 → `npx prettier --write <file>`
3. Python 文件 → `black <file>`（检测 black 是否可用）或 `ruff format <file>`
4. 以上都不可用 → 静默退出

退出码：始终 exit 0（格式化失败不阻断工作流，只 log 到 stderr）。

和现有 quality-gate.js 的关系：quality-gate 跑 linter（检查问题），auto-format 跑 formatter（修复格式）。两者用不同 matcher 或顺序执行：先 format 再 lint。

实现方式：在 hooks.json 的 PostToolUse 数组中，auto-format 排在 quality-gate 之前，确保先格式化再检查。

#### 3.2.2 typecheck.js（PostToolUse，matcher: Edit|Write|MultiEdit）

编辑 TypeScript 文件后做类型检查。

检测逻辑：
1. 文件扩展名不是 .ts/.tsx → 静默退出
2. 项目根目录无 tsconfig.json → 静默退出
3. 执行 `npx tsc --noEmit --pretty` 并捕获输出
4. 有类型错误 → 通过 respond() 返回错误摘要（截取前 500 字符）

退出码：始终 exit 0（类型错误作为信息反馈，不阻断）。

注意：tsc --noEmit 检查整个项目，可能较慢。设置 timeout 15 秒，超时则静默退出。

#### 3.2.3 commit-guard.js（PreToolUse，matcher: Bash）

提交代码前的质量校验。

触发条件：从 stdin 读取 tool_input.command，检测是否包含 `git commit`。不包含则立即 exit 0。

检查项（按顺序）：

1. **commit message 格式检查**：
   - 提取 commit message：支持 `-m "msg"` 直接参数和 `$(cat <<'EOF'...EOF)` heredoc 两种格式
   - 检查是否符合 `<type>: <description>` 格式（type 为 feat/fix/refactor/docs/test/chore/perf/ci）
   - 不符合 → stderr 输出建议格式，exit 2（阻止提交）

2. **console.log / print 检测**：
   - 对 staged 文件执行 `git diff --cached --diff-filter=ACM`
   - 在新增行中检测 `console.log`、`console.warn`、`console.error`（非注释行），以及 Python 的 `print(`
   - 发现 → stderr 提示哪些文件哪些行有 debug 输出，exit 2（阻止提交）

3. **密钥泄露检测**：
   - 对 staged 文件的新增行扫描以下模式：
     - API key 模式：`(api[_-]?key|apikey)\s*[:=]\s*['"][^'"]{10,}`
     - Token 模式：`(token|secret|password|passwd)\s*[:=]\s*['"][^'"]{8,}`
     - AWS 模式：`AKIA[0-9A-Z]{16}`
     - 私钥模式：`-----BEGIN (RSA |EC |DSA )?PRIVATE KEY-----`
   - 发现 → stderr 输出告警，exit 2（阻止提交）

退出码：
- exit 0：检查通过或命令不是 git commit
- exit 2：检查失败，阻止提交并将原因返回给 Claude

#### 3.2.4 block-no-verify.js（PreToolUse，matcher: Bash）

阻止跳过 git hooks。

触发条件：从 stdin 读取 tool_input.command，检测命令是否同时满足：
1. 包含 `git commit`（限定上下文，避免误匹配其他 git 子命令的 `-n`）
2. 包含 `--no-verify` 或 `-n` 标志

匹配到 → stderr 输出 "Blocked: --no-verify bypasses safety hooks. Remove the flag and commit normally."，exit 2。

## 六、技术约束

- **Windows 兼容**：所有 hook 脚本使用 Node.js（不依赖 bash/sh），路径处理兼容 `\` 和 `/`
- **无外部依赖**：hook 脚本只使用 Node.js 标准库 + 项目已安装的工具（prettier、biome、tsc 等），不引入新的 npm 包
- **超时保护**：每个 hook 有独立 timeout，超时静默退出不阻断工作流
- **幂等性**：/harvest 可重复运行，每次基于当前源数据生成，不依赖上次运行的状态
- **向后兼容**：v0.1.0 的所有功能保持不变，新增功能不破坏现有行为
