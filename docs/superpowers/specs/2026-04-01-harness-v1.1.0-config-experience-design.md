# sps-harness v1.1.0 — Config & Experience Upgrade

Date: 2026-04-01
Status: Draft
Scope: Project self-configuration, session experience, hook multi-language support

## Background

sps-harness v1.0.0 shipped with a complete hook/skill/agent/rule architecture. Two pain points remain:

1. **Low presence** — Users install the plugin but can't feel it working. Hooks run silently, session-start dumps a flat list of signals that Claude doesn't prioritize.
2. **Feature scatter** — 7 skills, 6 commands, 9 hooks. Users don't know which to use when.

Additionally, the project itself lacks a CLAUDE.md and SPEC.md, making development iterations harder. And the quality hooks (auto-format, typecheck, quality-gate) only support JS/TS, causing silent failures or wasted cycles in Python/Go/Rust projects.

## Goals

- Give sps-harness its own CLAUDE.md and SPEC.md for better dev experience
- Add allow whitelist to settings.json to reduce confirmation fatigue
- Make session-start output a single prioritized action instead of a flat list
- Add PostCompact hook to prevent command amnesia in long conversations
- Make quality hooks detect project type and run the right tools (JS/TS, Python, Go, Rust)

## Non-goals

- No new skills or commands
- No changes to agent definitions
- No changes to rule files
- No architectural changes to hook type system (Guard/Enhancer/Observer)

## Design

### 1. Project Self-Configuration

#### 1.1 CLAUDE.md (root)

New file. ~30 lines covering:

- Architecture overview (plugin structure: agents/, skills/, hooks/, rules/, commands/)
- Hook type system (Guard = can stop/prevent, Enhancer = inject context, Observer = record)
- Dev commands (no build step, CJS require, manual test by installing plugin)
- Code style (CommonJS, silent error handling, lib/utils.js shared functions)
- Gotchas (${CLAUDE_PLUGIN_ROOT} paths, timeout rules, settings.json deny vs hook-layer blocking)

Judgment criterion from the book: "Claude can derive from code → don't write; Claude can't guess → must write."

#### 1.2 SPEC.md (root)

New file. Records architectural decisions:

- Why Guard/Enhancer/Observer hook type split (determinism levels, different output channels)
- Why security-reviewer needs cross-session memory (false positive tracking, progressive trust)
- Plugin vs Superpowers responsibility boundary (no Planner/Generator/Evaluator here)
- workflow-map routing design rationale (stage-based, not feature-based)
- Why hooks are CJS not ESM (plugin runtime compatibility)

#### 1.3 settings.json allow whitelist

Add safe command pre-authorizations:

```json
{
  "permissions": {
    "allow": [
      "Bash(git status*)",
      "Bash(git log *)",
      "Bash(git diff *)",
      "Bash(git add *)",

      "Bash(npm test*)",
      "Bash(npx vitest *)",

      "Bash(rm -rf *)"
    ]
  }
}
```

### 2. session-start Upgrade

#### Current behavior

Detects 9 signals, outputs all matching ones as a numbered list via inject().

#### New behavior

Priority-ranked single action recommendation. Check conditions in order, output only the first match:

| Priority | Condition | Output |
|---|---|---|
| 1 | No .claude/ directory | "New project detected. Run /harness-audit for setup guidance." |
| 2 | Has design docs but no CLAUDE.md | "Design docs found but no CLAUDE.md. Run /harvest to generate." |
| 3 | Has Product-Spec but no code | "Ready for development. Start with brainstorming → writing-plans." |
| 4 | Security-sensitive files changed in last 3 commits | "Security-sensitive files changed. Consider /security-review." |
| 5 | Observations > 100 | "N operations recorded. Run /learn to discover patterns." |
| 6 | CLAUDE.md last updated > 30 days ago | "CLAUDE.md may be stale. Consider re-running /harvest." |
| 7 | Rules directory empty or stale (6+ months) | "Rules need attention. Run /rules to review." |

Removed from output (silent):
- Signal 4 (no test directory) — preference-level, too nagging for every startup
- Signal 6 (compact state exists) — handled by compact-restore hook already

Output format change:
- Before: `[sps-harness project status]\n1. ...\n2. ...\n3. ...`
- After: `[sps-harness] New project detected. Run /harness-audit for setup guidance.`

If all checks pass: no output at all (quiet protection).

File: `hooks/scripts/session-start.js`

### 3. PostCompact Hook

New hook. Fires after context compaction to re-inject essential command routing.

#### Injected content (~10 lines)

```
[sps-harness commands]
- /harvest — generate CLAUDE.md from design docs
- /harness-audit — assess .claude/ maturity
- /security-review — audit auth/payment/data code
- /save-compact — save state + compress context
- /learn — extract work patterns from operation records
- /rules — manage project rules
- /verify — adversarial verification of recent changes
- /compliance-review — compliance check
Route: security files changed → /security-review, design done no config → /harvest, periodic check → /harness-audit
```

#### hooks.json addition

```json
"PostCompact": [
  {
    "hooks": [
      { "type": "command", "command": "node \"${CLAUDE_PLUGIN_ROOT}/hooks/scripts/post-compact.js\"", "timeout": 3000 }
    ]
  }
]
```

Script logic: read static text, inject(). No conditional logic needed.

Files: `hooks/scripts/post-compact.js` (new), `hooks/hooks.json` (modify)

### 4. Hook Multi-Language Support

#### Project type detection

New shared function in `hooks/scripts/lib/utils.js`:

```javascript
function detectProjectType(cwd) {
  if (fs.existsSync(path.join(cwd, 'package.json')) || 
      fs.existsSync(path.join(cwd, 'tsconfig.json'))) return 'js';
  if (fs.existsSync(path.join(cwd, 'pyproject.toml')) || 
      fs.existsSync(path.join(cwd, 'requirements.txt'))) return 'python';
  if (fs.existsSync(path.join(cwd, 'go.mod'))) return 'go';
  if (fs.existsSync(path.join(cwd, 'Cargo.toml'))) return 'rust';
  return null;
}
```

Detection order = priority order. Monorepo with package.json + go.mod → JS wins.

#### Tool mapping

| Project type | auto-format | typecheck | quality-gate (lint) |
|---|---|---|---|
| js | prettier or eslint --fix | tsc --noEmit | eslint |
| python | ruff format | mypy | ruff check |
| go | gofmt | go vet | golangci-lint |
| rust | cargo fmt | cargo check | cargo clippy |
| null | skip | skip | skip |

#### Rules

- Before calling any tool, check if it exists (`which <tool>` / `where <tool>` on Windows). Missing tool → silent skip.
- Unrecognized project type → silent skip, no error, no output.
- Each hook script calls `detectProjectType(cwd)` at the top, selects tool, runs it.

Files modified:
- `hooks/scripts/lib/utils.js` — add detectProjectType + tool existence check helper
- `hooks/scripts/auto-format.js` — use detectProjectType, branch by type
- `hooks/scripts/typecheck.js` — use detectProjectType, branch by type
- `hooks/scripts/quality-gate.js` — use detectProjectType, branch by type

## Risk

- **Tool detection false positive**: monorepo with multiple config files could pick wrong type. Mitigation: priority order is documented, user can override later.
- **Missing tools**: Python/Go/Rust users may not have ruff/mypy/golangci-lint installed. Mitigation: silent skip when tool not found.
- **allow whitelist too broad**: `Bash(npm run *)` allows any npm script including destructive ones. Acceptable because npm scripts are user-defined and expected.
