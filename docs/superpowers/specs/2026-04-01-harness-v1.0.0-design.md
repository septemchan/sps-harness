## 1. 背景与目标

### 为什么做这次升级

2026-03-31 Claude Code 源码泄露后，通过 instructkr/claw-code 和 Xiao Tan 的深度研究报告，我们获得了 Claude Code 引擎层的完整架构认知。这些发现暴露了 harness 的外壳层（rules/hooks/skills）和引擎层之间的对齐差距。

### 核心发现

1. **Hook 能力严重低估**：harness 的 hooks 几乎只用了 `respond()` 返回 message，但 Claude Code 的 hook 协议支持 7 种返回能力（permissionBehavior, updatedInput, additionalContext, preventContinuation, stopReason, blockingError, message）
2. **缺少"模型可感知"机制**：模型不知道 harness 有什么能力、当前项目该用哪个
3. **缺少对抗式验证**：verification-loop 是"确认没挂"心态，不是"想办法找到问题"心态
4. **缺少规范合规审查**：superpowers 的 code-reviewer 不感知 harness 的 rules

### v1.0.0 目标

将 harness 从"建议型辅助工具"升级为"治理型工程框架"，与 Claude Code 引擎层的设计意图对齐。

---

## 2. 改动总览

| 方向 | 改动 | 大白话 |
|------|------|--------|
| Hook 架构升级 | 全部 16 个 hooks 审查，重新分配角色 | 以前只是提醒你，现在该挡的挡、该塞信息的塞信息 |
| session-start 动态注入 | 扫描项目状态，注入动态引导 | 开对话时自动告诉模型"项目现在什么状态、该干嘛" |
| Verification Agent | 删除 verification-loop skill，新建对抗式 agent | 从"跑检查确认没挂"变成"想尽办法找问题" |
| Compliance Reviewer Agent | 新增 harness-aware 合规审查 agent | 对照你定的 4 套规范检查代码是否合规 |

## 3. Hook 架构升级

### 3.1 设计原则

按 Claude Code 14 步 tool 执行管线，每个 hook 分配为三种角色之一：

| 角色 | 核心能力 | 大白话 |
|------|--------|--------|
| **Guard（守卫）** | `permissionBehavior: "deny"` / `blockingError` / `stopReason` | 直接挡住不让做 |
| **Enhancer（增强）** | `additionalContext` / `updatedInput` / `preventContinuation` | 塞信息给模型，让它下一轮做得更好 |
| **Observer（观察）** | `message`（保持现状） | 只记录不干预 |

### 3.2 utils.js 升级

在现有 `respond()` 基础上新增语义化工具函数：

```javascript
// 现有（保留）
respond(message)

// 新增
deny(reason)               // 守卫：直接阻止
inject(context, message?)  // 增强：注入上下文（可选附带 message）
prevent(reason)            // 增强：不 deny 但阻止流程继续
stop(reason)               // 守卫：控制停止原因
guard(condition, denyReason, passContext?)  // 条件守卫：满足条件 deny，否则注入 context
```

输出格式统一为 JSON，通过 `process.stdout.write(JSON.stringify(...))` 输出。

### 3.3 全部 16 个 Hooks 的角色分配

#### SessionStart hooks（2 个）

**session-start → Enhancer**
- 以前：检查 CLAUDE.md 有没有，发 message 提示
- 现在：扫描项目状态（设计文档、Product-Spec.md、app/ 代码、测试、observations、安全文件变动），组装动态引导，通过 `inject()` 注入 additionalContext
- 大白话：开对话时自动告诉模型"你现在处于什么阶段、推荐用什么"

扫描信号：

| 信号 | 检测方式 | 注入的引导 |
|------|--------|----------|
| 有设计文档，无 CLAUDE.md | 扫描 docs/superpowers/specs/ | 推荐 /harvest |
| 有 Product-Spec.md，无 app/ 代码 | 检查根目录 | 产品规格就绪，该进入开发 |
| 有 app/ 代码，无测试文件 | 扫描 app/ 和 test/ | 缺测试，推荐 TDD 流程 |
| observations.jsonl 超 100 条 | 检查文件行数 | 该跑 /learn 提取模式 |
| .compact/state 文件存在 | 检查 .compact/ | 上次压缩过，可能需要恢复上下文 |
| 安全相关文件最近有改动 | git diff 检查 auth/payment/user 路径 | 建议跑 /security-review |

**compact-restore → Enhancer**
- 以前：读 state 文件，发 message 告诉模型
- 现在：将恢复的上下文通过 `inject()` 注入，模型能直接"看到"压缩前的工作状态
- 大白话：压缩恢复时，不只是提醒"之前在干嘛"，而是把状态直接塞进上下文

#### PreToolUse hooks（3 个）

**block-no-verify → Guard**
- 以前：用 process.exit(1) 阻止
- 现在：改用 `deny()` 输出 `permissionBehavior: "deny"` + 明确 reason
- 大白话：功能不变，但用正式的 hook 协议阻止，而不是粗暴退出

**commit-guard → Guard + Enhancer 混合**
- 以前：检测问题发 message 建议
- 现在：三种情况三种处理
  - 检测到密钥 → `deny("检测到疑似密钥，禁止提交")` 直接阻止
  - 检测到 console.log → `inject("发现 console.log 在 file:line，请清理后再提交")` 告知模型具体位置
  - commit message 格式错误 → `inject("commit message 不符合 Conventional Commits 格式，建议修改为: ...")` 给出修正建议
- 大白话：发现密钥直接挡住（以前只是提醒）；发现 console.log 告诉模型在哪（以前只说"有"）

**suggest-compact → Enhancer**
- 以前：计数到 ~50 发 message 提醒
- 现在：改用 `inject()` 注入压缩建议 + 当前工具调用计数
- 大白话：压缩建议变成上下文的一部分，模型能据此判断是否需要 /save-compact

#### PostToolUse hooks（8 个）

**auto-format → Enhancer**
- 以前：跑格式化，发 message
- 现在：格式化有改动时，用 `inject()` 注入改动摘要（哪些文件被格式化了）
- 大白话：模型知道哪些文件被自动格式化了，不需要再手动处理

**typecheck → Enhancer + prevent**
- 以前：跑 tsc 发 message 告知有错误
- 现在：有类型错误时，用 `inject()` 注入完整错误列表 + `prevent()` 阻止流程继续
- 大白话：类型错误不再是"提醒"，而是"必须先修了才能继续"

**quality-gate → Enhancer**
- 以前：建议"去跑 linter"
- 现在：直接跑 linter 并将结果通过 `inject()` 注入
- 大白话：不再建议你去做，而是帮你做了并告诉模型结果

**observe → Observer（不改）**
- 纯观察记录，不干预
- 大白话：只记录，不管

**ui-prompt → Enhancer**
- 以前：条件满足时发 message 建议
- 现在：改用 `inject()` 注入建议
- 大白话：建议成为上下文的一部分，不是一闪而过的提示

**launch-prompt → Enhancer**
- 同 ui-prompt 改法

**spec-sync-prompt → Enhancer**
- 同 ui-prompt 改法

**check-prompt → Enhancer**
- 同 ui-prompt 改法

#### Stop hooks（1 个）

**completion-guard → Guard**
- 以前：检测安全敏感文件，发 message 建议跑 /security-review
- 现在：检测到安全敏感文件编辑时，用 `stop()` 阻止完成 + `inject()` 注入"必须先跑 /security-review"
- 大白话：碰了安全文件就不让结束，必须先做安全审查（以前只是建议）

#### PreCompact hooks（1 个）

**pre-compact → Observer（不改）**
- 纯状态保存，不干预
- 大白话：只存状态，不管

### 3.4 命令全覆盖：每个命令都有 hook 自动建议

设计原则：**用户不需要记命令，hook 在对的时机提醒。**

| 命令 | 哪个 hook 建议 | 触发条件 |
|------|-------------|--------|
| `/harvest` | session-start | 有设计文档但无 CLAUDE.md |
| `/harness-audit` | session-start | 无设计文档也无 CLAUDE.md |
| `/security-review` | completion-guard + session-start | 编辑了安全文件 / 安全文件最近有改动 |
| `/save-compact` | suggest-compact | ~50 次工具调用 |
| `/learn` | session-start | observations 超 100 条 |
| `/rules` | session-start（扩展） | rules/ 目录为空，或规则文件超过 6 个月未更新 |
| `/verify` | completion-guard（扩展） | 本次会话有显著代码变更（多文件编辑），结束前建议先验证 |
| `/compliance-review` | commit-guard（扩展） | 提交前，如果本次会话还没跑过 /compliance-review，注入建议 |

**扩展改动（在现有 hook 里加条件分支，不新增脚本）：**

**completion-guard 扩展：**
- 以前：只检测安全敏感文件
- 现在：增加一条逻辑，当检测到本次会话有多文件代码变更时，用 `inject()` 建议先跑 `/verify` 再结束

**commit-guard 扩展：**
- 以前：检测密钥、console.log、commit message 格式
- 现在：增加一条逻辑，提交前检查本次会话是否跑过 `/compliance-review`，没跑过则 `inject()` 建议先跑

**session-start 扩展：**
- 以前（v1.0.0 新设计）：已包含 6 个扫描信号
- 现在：增加一条信号，检测 rules/ 目录为空或 `.md` 文件的修改时间超过 6 个月，建议跑 `/rules` 检查

### 3.5 角色统计

| 角色 | 数量 | Hooks |
|------|------|-------|
| Guard | 3 | block-no-verify, completion-guard, commit-guard（密钥检测时） |
| Enhancer | 11 | session-start, compact-restore, suggest-compact, auto-format, typecheck, quality-gate, ui-prompt, launch-prompt, spec-sync-prompt, check-prompt, commit-guard（非密钥时） |
| Observer（不改） | 2 | observe, pre-compact |

## 4. Verification Agent

### 4.1 为什么从 skill 改为 agent

| 维度 | skill（fork 模式） | agent |
|------|-------------------|-------|
| context | 共享主线程 context（fork） | 独立 context（不受主线程"我做完了"的 bias 影响） |
| token budget | 共享 | 独立（不和主线程抢） |
| 工具权限 | 继承全部 | 可限制（只给验证需要的） |
| 持久性 | 一次性 | 可跨会话记忆 |

### 4.2 Agent 定义

```yaml
name: verification-agent
description: >
  对抗式验证 agent。心态不是"确认没问题"，而是"try to break it"。
  独立于实现者的 context，避免验证偏差。
  对照 harness rules 做规范合规检查。
model: sonnet
maxTurns: 30
allowedTools:
  - Read
  - Glob
  - Grep
  - Bash
```

### 4.3 Prompt 核心设计

**角色定位：**
你是对抗式验证员。你的工作不是确认实现没问题，而是 try to break it。

**两类失败模式警告（开头就说）：**
1. **Verification avoidance** — 只看代码不跑检查、不跑命令就写 PASS
2. **80% 迷惑** — UI 看起来还行、测试也过了，就忽略最后 20% 的问题

**强制检查清单：**
- build（必须跑，记录输出）
- test suite（必须跑，记录输出）
- linter / type-check（必须跑，记录输出）
- 根据变更类型做专项验证：
  - frontend → 浏览器自动化 / 页面子资源验证
  - backend → curl/fetch 实测响应
  - CLI → stdout/stderr/exit code
  - migration → up/down 和已有数据
  - refactor → public API surface
- **adversarial probes**（必须做）：主动尝试边界情况、异常输入、权限绕过
- 每个 check 必须带 **command** 和 **output observed**
- 读取 harness rules（coding-standards, testing-standards, security-standards, git-standards），对照做基本合规检查（注意：这里的合规检查是验证过程中的附带检查，和 compliance-reviewer 的专门深度审查不同）

**输出格式：**
最后必须输出：

```
## VERDICT: PASS | FAIL | PARTIAL

### Checks performed
- [PASS/FAIL] check name — command: `xxx` — output: ...

### Violations found
- [VIOLATION] rule reference — file:line — description

### Adversarial probes
- [PROBE] description — result
```

### 4.4 清理

- 删除：`skills/verification-loop/`（整个目录）
- 删除：对应 command（如有 `commands/verification-loop.md`）
- 新增：`agents/verification-agent.md`
- 新增：`commands/verify.md`
- 更新：`rules/workflow-map.md` 中 verification-loop 的引用

## 5. Compliance Reviewer Agent

### 5.1 与 superpowers code-reviewer 的分工

| 维度 | superpowers code-reviewer | harness compliance-reviewer |
|------|--------------------------|----------------------------|
| 看什么 | 代码质量、架构、可读性 | 规范合规性（对照 4 套 rules） |
| 基于什么 | 通用最佳实践 | 项目的具体规则文件 |
| 什么时候用 | 大功能完成后整体审查 | 每次提交前快速合规检查 |
| 命名 | code-reviewer | compliance-reviewer |

### 5.2 Agent 定义

```yaml
name: compliance-reviewer
description: >
  规范合规审查 agent。对照 harness 的 4 套 rules
  （coding-standards, testing-standards, security-standards, git-standards）
  检查代码是否合规。不做通用代码审查，只做规范对照。
model: sonnet
maxTurns: 30
allowedTools:
  - Read
  - Glob
  - Grep
  - Bash
```

Bash 限制为只读命令：git diff, git log, git show, git blame 等。

### 5.3 Prompt 核心设计

**角色定位：**
你是规范合规审查员，不是通用 code reviewer。你的唯一职责是对照 harness rules 检查代码合规性。

**工作流程：**
1. 第一步必须读取 harness 的 4 个 rules 文件：
   - `rules/coding-standards.md`
   - `rules/testing-standards.md`
   - `rules/security-standards.md`
   - `rules/git-standards.md`
2. 获取审查范围（默认：最近一次 commit 的 diff，可通过参数扩展到 branch diff）
3. 逐条对照 rules 检查

**对每条发现，必须包含：**
- 严重级别：`VIOLATION`（必须修）/ `WARNING`（建议修）/ `SUGGESTION`（可选优化）
- 引用的具体 rule 条目（如 "coding-standards: 函数不超过 50 行"）
- 文件位置（file:line）
- 简短说明

**输出格式：**

```
## Compliance Score: X/Y rules passed

### VIOLATIONS (must fix)
- [coding-standards] file:line — 函数 processData() 82 行，超过 50 行限制

### WARNINGS (should fix)
- [testing-standards] src/api/ — 新增 3 个 API 端点，未发现对应测试文件

### SUGGESTIONS (nice to have)
- [git-standards] commit abc123 — message "update" 不符合 Conventional Commits，建议 "feat: ..."
```

### 5.4 新增文件

- 新增：`agents/compliance-reviewer.md`
- 新增：`commands/compliance-review.md`
- 更新：`rules/workflow-map.md` 中加入 compliance-review 的触发时机
