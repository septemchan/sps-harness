# Harvest

Generate `.claude/CLAUDE.md` so Claude has project context in every session.

## Two modes

**Mode A — Design docs exist** (`docs/superpowers/specs/` has files):
1. Read the most recent spec file in `docs/superpowers/specs/`
2. Also read `docs/superpowers/plans/` if present
3. Scan project files (package.json, tsconfig.json, requirements.txt, go.mod, pyproject.toml) for commands and dependencies
4. Scan `src/` directory structure for architecture overview
5. Generate CLAUDE.md combining spec insights + project file data

**Mode B — No design docs** (fallback):
1. Scan project files only:
   - `package.json` → scripts (commands), dependencies (tech stack)
   - `tsconfig.json` / `pyproject.toml` / `go.mod` → language config
   - `src/` or root directory structure → architecture
   - `.env.example` or `.env.local` → environment variable names (keys only, never values)
2. Generate CLAUDE.md from inferred data
3. Add comment: `<!-- Generated by /harvest (inferred mode). Review and refine. -->`

## Output structure

Generate `.claude/CLAUDE.md` with exactly these sections:

```
# [Project Name]

## 技术栈
[Framework] + [Database] + [Key dependencies]

## 命令
[dev command]       # Development server
[build command]     # Production build
[test command]      # Run tests
[lint command]      # Code checks

## 架构
[One paragraph describing overall architecture]

[Directory tree with annotations for each major directory]

## 约定
- [Convention 1: file organization]
- [Convention 2: data access patterns]
- [Convention 3: other project-specific conventions]

<!-- Generated by /harvest from [source]. Last updated: YYYY-MM-DD -->
```

Keep the output under 100 lines total.

## Update behavior

If `.claude/CLAUDE.md` already exists:
1. Read the existing file
2. Generate a new version from current sources
3. Compare the two versions and show the diff to the user
4. Ask the user: accept all updates / select by section / cancel
5. Preserve any sections with headings NOT in the generated template (技术栈, 命令, 架构, 约定) — these are user-added custom sections
6. Update the bottom comment with current date

## Workflow

1. Detect mode (check for `docs/superpowers/specs/`)
2. Collect information (read docs + scan files)
3. Check for existing `.claude/CLAUDE.md`
4. Generate CLAUDE.md content
5. If first time: write directly. If update: show diff, ask for confirmation
6. Ensure `.claude/` directory exists, write the file
7. Verify: confirm the file is under 100 lines
8. Stage and commit the generated CLAUDE.md

## 已知问题

- monorepo 根目录的 package.json 可能包含 workspace 管理工具而非实际技术栈，容易将 devDependencies（如 turbo、lerna）误认为核心依赖
- 生成的 CLAUDE.md 在依赖较多的项目中容易超过 100 行限制，需要在生成后主动检查并精简
- `.env.example` 中的注释行有时会被误读为环境变量名
- 当 `src/` 目录结构很深时，架构概览倾向于过度展开子目录，应只展示前两层

## Important

- Never read .env file values — only read key names from .env.example or .env.local
- Use Chinese for section headings (技术栈, 命令, 架构, 约定) to match the user's preference
- The CLAUDE.md describes the project as-is, not aspirational goals
- If information cannot be determined, omit that section rather than guessing
